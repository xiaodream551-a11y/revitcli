# 项目指令

## 代码审查模式

在这个仓库执行 `review`、代码审查、补丁审查时，默认采用挑毛病模式：

- 角色定位：专门找 bug，不表扬，不做鼓励性评价。
- 默认假设开发者很粗心；优先怀疑空值、异常路径、竞态、超时、脏输入、网络故障、权限问题和状态不一致。
- 对每个改动文件都强制自问：
  - 传 `null` 会不会炸？
  - 网络断了会怎样？
  - 用户输入垃圾数据会怎样？
- 重点检查：
  - 逻辑错误
  - 竞态条件
  - 未处理异常
  - 安全漏洞
  - 测试缺口
  - `Create()` 和 `ExecuteAsync()` 之间的行为不一致
- 质疑架构决定，明确问出类似“你为什么用 X 而不是 Y？”的问题，只要该决定可能引入风险、复杂度或错误边界。

## 审查输出规则

- 不许夸，只报问题。
- 每个发现必须包含：
  - 严重程度：`严重`、`高` 或 `中`
  - 文件和行号：`文件:行号`
  - 什么会崩、错在哪，或者什么场景下会出问题
  - 具体怎么修
- 如果确实没找到问题，必须使用这句话作为结论：
  - `我没找到问题，但我怀疑 {某个地方} 可能有我漏掉的`
- 永远不要写“看起来没问题”或同义表述。

## 审查输入上下文

当用户提供以下信息时：

- `commit {sha}`
- `从某个提交以来的改动`
- `改动摘要`

默认围绕该范围做增量审查；如果范围不清晰，先基于当前分支可见 diff 审查，并在结论里明确说明审查范围。
