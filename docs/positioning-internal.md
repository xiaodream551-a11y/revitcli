### 为什么从"不做 AI-to-Revit 桥梁"翻转回 agent 方向

旧 vision 的逻辑是"社区 MCP server 已覆盖 AI 操作 Revit 场景"。但这个判断有缺陷:

1. **社区 MCP server 是个人玩具级**,没有 audit、没有合规、没有团队治理,**企业生产场景不能用**
2. **"AI 操作 Revit"和"agent + BIM 工程师协作"是两件事** — 前者是 AI 替代,后者是 AI 协助。社区只覆盖了前者
3. **真实用户(用户本人)就是会用 agent 的 BIM 工程师**,这个画像不应该被 vision 否定

新定位:**继续做 BIMOps 运行器(已有立足点),同时把 agent-native 升级为长期主线**。两轨在路线图上自然合流——A1 / A5 嵌入 v1.5,A2 嵌入 v1.6,A3 嵌入 v1.7。

## 二、反派叙事:明确不讲什么

这些路径会让叙事崩,**永久禁用**:

| #   | 不讲                         | 为什么                                                        |
| --- | ---------------------------- | ------------------------------------------------------------- |
| 1   | "AI 替代 BIM 工程师"         | 工程师是用户,得罪用户的工具不会被用                           |
| 2   | "全自动 BIM 设计"            | 过度承诺,设计判断 LLM 短期做不到;一旦说大话 demo 失败就被钉死 |
| 3   | "我们比 Autodesk 强"         | 个人 vs 巨头是软肋,不是 angle;Autodesk 真要做能秒杀           |
| 4   | "通用 AI for BIM"            | 太泛,听众不知道你具体做什么                                   |
| 5   | "纯命令行工具"               | 限制想象空间,还会让非技术受众跑掉                             |
| 6   | 从 6 个功能讲起              | 听众会迷失;永远从协议栈或三柱切入                             |
| 7   | "MCP 替代品"                 | MCP 是协议,我们是协议的一种实现 + 上层工具,不是替代           |
| 8   | "无代码 / 零门槛"            | RevitCli 用户必然懂命令行;假装零门槛会吸引错的用户群          |
| 9   | "BIM 革命"                   | 革命叙事在 BIM 行业被滥用,听众已麻木                          |
| 10  | 跟 Dynamo / Grasshopper 对标 | 那些是可视化编程工具,跟 agent-native 不在一个层               |

### 危险词汇黑名单

- ❌ "革命性" / "颠覆" / "全新一代"
- ❌ "替代人类" / "无需工程师"
- ❌ "All-in-one" / "一站式" — 模糊定位
- ❌ "AI 驱动" — 几乎所有产品都这么说,无信息量

### 替换词汇

- 用 "**agent-native**" 替代 "AI-powered" — 更精确
- 用 "**协议层 / protocol layer**" 替代 "平台 / platform" — 更技术
- 用 "**without giving up control**" 替代 "安全可靠" — 更具体
- 用 "**for BIM engineers who use agents**" 替代 "for everyone" — 受众清晰

## 三、自我决策树:下一步该做什么

按当前最想解决的问题分支选择。**每条分支都有进入条件 + 退出条件**。

### 分支 A:我自己做下一步选择时方向不清

```
进入条件:每次想加功能都纠结"这是不是项目该做的"
↓
做法:把三柱叙事(可重复 / 可信任 / 可学习)内化为决策标尺
↓
每个新功能问三遍:它属于哪一柱?如果不属于任何一柱,默认不做
↓
退出条件:连续 3 次新功能决策都能用三柱秒判
```

### 分支 B:发个东西让圈子里的人知道

```
进入条件:有想发的渠道(知乎 / 掘金 / Hacker News / X / 微信群)
↓
做法:用 narrative.md 样板 B(中文)或样板 A(英文)
↓
配套:1 张协议栈图 + 1 个 30 秒 demo gif(record terminal session)
↓
退出条件:发出 + 收到 ≥10 条具体反馈(不是泛泛点赞)
↓
反馈不够:不是渠道问题就是叙事问题,**回头改本文 §二的反派叙事**
```

### 分支 C:让别人贡献代码

```
进入条件:有 ≥3 个独立用户(不是朋友)用过且愿意 issue
↓
做法:
  - CONTRIBUTING.md 加 narrative.md 链接
  - 把"good first issue"标签贴 5-10 个 docs / completions / starter profile 类小活
  - 在 README 显眼位置写"Why this exists"(用样板 A 第一段)
↓
退出条件:第一个不认识的人开了 PR
```

### 分支 D:找合作伙伴 / 小团队

```
进入条件:个人精力撑不住或想加速
↓
做法:
  - 准备样板 C(电梯路演)
  - 3 张 slide:协议栈图 + 三柱 + milestone 路线图
  - 想清楚"我能给合伙人什么 + 他能给我什么"
↓
退出条件:接到第一次正式合伙意向 / 拒绝
↓
被拒:大概率叙事 OK 但 traction 不够,回到分支 B 攒数据
```

### 分支 E:思考是不是值得长期投入

```
进入条件:做了一段时间,开始怀疑值不值得
↓
做法:**不要写文档,做用户访谈**
  - 找 5-10 个会用 agent 的 BIM 工程师(微信 / LinkedIn / 知乎私信)
  - 念样板 C,问三个问题:
    1. 听完你能复述一遍我是干什么的吗?
    2. 这解决你什么具体痛点?
    3. 你愿意试试吗?为什么不?
  - **不要 demo**,只要听他们的话
↓
退出条件:5 个访谈完成
↓
结果:
  - 80%+ 能复述 + 说出真实痛点 → 继续,加速
  - 50% 能复述但说不出痛点 → 叙事对但市场小,降低投入或转 niche
  - <30% 能复述 → 叙事还没对,回本文 §二打磨
```

## 四、不同场景的"该说什么 / 不该说什么"卡片

### 场景 1:陌生人在群里问"这是干什么的"

✅ 说:"agent + BIM 工程师协作的命令行工具,让 agent 帮工程师跑繁琐流程,有 audit trail。"
❌ 不说:列功能、讲架构、提 Autodesk

### 场景 2:工程师朋友问"我怎么用"

✅ 说:具体场景(出图清单、参数批量改、CSV 导入),给一条命令
❌ 不说:"协议层"、"三柱"、技术术语

### 场景 3:技术博客 / 公开演讲

✅ 说:协议栈、agent-native、对标 LSP/MCP、技术新颖性
❌ 不说:个人故事(除非有共鸣点)、过度营销

### 场景 4:招贡献者

✅ 说:开源、可扩展、第一个 agent-native BIM 协议、有路线图
❌ 不说:"还没成熟"、"我也不确定方向"(诚实但劝退)

### 场景 5:跟用户的老板讲

✅ 说:audit trail、合规、知识资产、新人 onboarding
❌ 不说:agent、AI、技术细节

- [ ] 30 秒 elevator pitch 还能脱口而出吗?

- [ ] 反派词汇是否在最近的 README / 提交信息里悄悄出现?

## 六、最强叙事路径(理想情况,标代价)

如果资源全开,做完整的 **Brand Story 体系**:

1. **基础层(已做)**:`docs/narrative.md` 作为素材库
2. **传播层**:改写 README 用样板 A,加协议栈图
3. **深度内容**:写一篇 2000-3000 字博客文章,投知乎/Hacker News
4. **演示层**:录 5 分钟 demo 视频(场景:工程师周五下午被改单 → agent 干完)
5. **品牌层**:6 张 slide deck(协议栈 / 三柱 / milestone / 用户证言空位 / 路线 / 联系方式)
6. **入口层**:注册 landing page(revitcli.dev 或类似),挂演示视频
7. **验证层**:找 5-10 个真实 BIM 工程师做用户访谈,收 3-5 句证言

**代价**:每步 0.5-2 天,加起来 10-15 天工作量。**做完不一定有用**——前提是用户群存在且能找到。

**推荐起点(最低成本验证)**:

1. ✅ 落盘 narrative.md(已完成)
2. 改 README 第一段(0.5h)
3. 找 3 个会用 agent 的 BIM 工程师朋友,把样板 C 念给他们听,看反应(1 周)

反应分级:

- "这正是我要的!" → 加速,启动深度博客 + landing page
- "哦不错" → 继续按路线图推
- "听不懂" → **回本文 §二、§三 重打磨叙事**

## 七、Open Questions(留给后续讨论)

1. **要不要做一个 landing page?** — 有助于 SEO 和分享,但维护成本不低,且没用户基数时空架子尴尬
2. **要不要在 GitHub 加 Sponsors 或商业模式?** — 当前个人项目无所谓,如果要团队化必须先想
3. **MCP server 包装(侧线 Track)什么时候做?** — narrative 里讲了"我们是协议层",但还没有 MCP server 实现。先做完 v1.5 再做?还是优先做以兑现叙事?
4. **要不要给 Autodesk 主动联系?** — 长期可能是合作或被收购方;过早接触可能被复制
5. **多语言支持(中英)如何分工?** — 当前 README 英文 + 注释中英混合,叙事样板分了中英,但没明确哪些场景用哪个语言
